📚 المحاضرة الأولى: الاستعلامات المتداخلة (Subqueries)

🔑 الملخص الشامل

الـ Subquery هو استعلام SELECT داخل استعلام SELECT آخر. الاستعلام الخارجي يُسمى Outer Query والداخلي يُسمى Inner Query. يُستخدم في SELECT list، WHERE clause، وحتى مع DELETE وUPDATE.

1️⃣ ما هي الـ Subquery؟

🇮🇶 العربية
الـ Subquery هي استعلام SQL مُدمج داخل استعلام آخر. الاستعلام الخارجي (Outer Query) هو الاستعلام الرئيسي، والاستعلام الداخلي (Inner Query) هو الذي يكون بداخله بين قوسين. يمكن أن يحتوي الاستعلام الداخلي على WHERE و HAVING و GROUP BY.
🇬🇧 ENGLISH
A Subquery is a SELECT statement nested inside another SELECT statement. The main query is called the Outer Query, and the nested one is the Inner Query. It must be enclosed in parentheses and can contain WHERE, HAVING, and GROUP BY clauses.

2️⃣ Subquery في SELECT List

يجب أن يُرجع الاستعلام الداخلي قيمة واحدة فقط (Scalar Subquery). مثال:

SELECT Category,
       (SELECT MAX(DVDPrice) FROM Films
        WHERE Films.CategoryId = Category.CategoryId),
       CategoryId
FROM Category;
شرح السطر 2-4: الاستعلام الداخلي يعمل لكل صف في جدول Category ويُرجع أعلى سعر DVD لتلك الفئة. هذا يُسمى Correlated Subquery لأنه مرتبط بالاستعلام الخارجي.

3️⃣ Subquery في WHERE Clause

SELECT Category, FilmName, DVDPrice
FROM Category INNER JOIN Films
    ON Category.CategoryId = Films.CategoryId
WHERE Films.DVDPrice =
    (SELECT MIN(DVDPrice) FROM Films
     WHERE Films.CategoryId = Category.CategoryId);
شرح السطر 1: نختار اسم الفئة، اسم الفيلم، والسعر.
السطر 2-3: ندمج جدول Category مع Films.
السطر 4-6: الشرط: سعر الـ DVD يجب أن يساوي أدنى سعر لنفس الفئة — يُرجع أرخص DVD لكل فئة.

4️⃣ مشغلات الـ Subquery

🔹 IN Operator

SELECT FirstName, LastName, YEAR(DateOfBirth)
FROM MemberDetails
WHERE YEAR(DateOfBirth) IN
    (SELECT YearReleased FROM Films);
شرح يُرجع الأعضاء الذين وُلدوا في نفس سنة إصدار أي فيلم. الـ IN يتحقق إذا كانت القيمة موجودة في قائمة النتائج.

🔹 ANY / SOME Operator

SELECT FirstName, LastName
FROM MemberDetails
WHERE DateOfJoining < ANY
    (SELECT MeetingDate FROM Attendance);
شرح ANY تعني: الشرط يصح إذا تحقق مع أي قيمة واحدة على الأقل من نتائج الاستعلام الداخلي. (= ANY يساوي IN)

🔹 ALL Operator

SELECT MemberId
FROM MemberDetails
WHERE MemberId < ALL
    (SELECT FilmId FROM Films WHERE FilmId > 5);
شرح ALL تعني: الشرط يجب أن يصح مع كل القيم. هنا: MemberId يجب أن يكون أصغر من كل الـ FilmIds التي قيمتها أكبر من 5.

🔹 EXISTS Operator

SELECT City
FROM Location
WHERE EXISTS
    (SELECT * FROM MemberDetails
     WHERE MemberId < 5);

-- عكسه بـ NOT EXISTS:
SELECT City FROM Location
WHERE NOT EXISTS
    (SELECT * FROM MemberDetails
     WHERE MemberId > 99);
شرح EXISTS لا تقارن قيماً بل تتحقق فقط هل الاستعلام الداخلي أرجع صفوفاً أم لا. إذا أرجع أي صف → TRUE → يُنفذ الاستعلام الخارجي.
⚠️ ملاحظة مهمة: يمكن استخدام Subquery مع DELETE وUPDATE في الـ WHERE clause.

🏗️ المحاضرة الثانية: لغة تعريف البيانات (DDL)

🔑 الملخص الشامل

DDL = Data Definition Language. تُستخدم لإنشاء وتعديل وحذف هياكل قاعدة البيانات (جداول، قيود، فهارس). الأوامر الرئيسية: CREATE، ALTER، DROP. القيود الرئيسية: NOT NULL، UNIQUE، CHECK، PRIMARY KEY، FOREIGN KEY.

1️⃣ إنشاء وحذف قاعدة البيانات

-- إنشاء قاعدة بيانات
CREATE DATABASE MyDatabase;

-- حذف قاعدة بيانات
DROP DATABASE MyDatabase;

2️⃣ أنواع البيانات (Data Types)

CHAR(n)
نص بطول ثابت
Fixed-length string. Always uses exactly n characters, padded with spaces if shorter.
VARCHAR(n)
نص بطول متغير
Variable-length string up to n characters. Saves space compared to CHAR.
NUMBER / INT
أرقام
Stores integers or fixed/floating point numbers.
DATE
تاريخ ووقت
Stores date and time information.
AUTOINCREMENT
عداد تلقائي
Used as an integer counter that auto-increments with each new row.

3️⃣ إنشاء الجداول (CREATE TABLE)

CREATE TABLE Train_Times
(
    start_location  varchar(75),
    destination     varchar(75),
    departs         time,
    arrives         time
);
شرح السطر 1: إنشاء جدول اسمه Train_Times.
السطر 3-6: تعريف 4 أعمدة: موقع الانطلاق، الوجهة، وقت المغادرة، وقت الوصول.

4️⃣ تعديل الجداول (ALTER TABLE)

-- إضافة عمود
ALTER TABLE MyTable
ADD new_column varchar(50);

-- حذف عمود
ALTER TABLE MyTable
DROP COLUMN old_column;

-- تعديل نوع بيانات (SQL Server)
ALTER TABLE MyTable
ALTER COLUMN my_col int;

-- في Oracle و MySQL تستخدم MODIFY
ALTER TABLE MyTable
MODIFY my_col int;

5️⃣ القيود (Constraints)

🔸 NOT NULL

CREATE TABLE MyTable
(
    Column1 int NOT NULL,      -- يجب أن يكون له قيمة
    Column2 varchar(20),         -- يمكن أن يكون NULL
    Column3 varchar(12) NOT NULL
);
شرح NOT NULL يمنع إدخال قيم فارغة في العمود. Column2 يسمح بـ NULL (فارغ)، أما Column1 و Column3 فلا بد أن يكون لهما قيمة.

🔸 UNIQUE

-- الطريقة 1: في تعريف العمود مباشرة
CREATE TABLE MyUniqueTable
(
    Column1 int,
    Column2 varchar(20) UNIQUE,
    Column3 varchar(12) UNIQUE
);

-- الطريقة 2: بإعطائها اسم (تسمح بـ UNIQUE على عمودين معاً)
CREATE TABLE AnotherTable
(
    Column1 int,
    Column2 varchar(20),
    Column3 varchar(12),
    CONSTRAINT MyUniqueConstraint UNIQUE(Column2, Column3)
);

-- إضافة بعد الإنشاء:
ALTER TABLE YetAnotherTable
ADD CONSTRAINT MyUniqueConstraint UNIQUE(Column2, Column3);

-- حذف القيد:
ALTER TABLE YetAnotherTable
DROP CONSTRAINT MyUniqueConstraint;

-- في MySQL:
ALTER TABLE YetAnotherTable
DROP INDEX MyUniqueConstraint;
شرح UNIQUE يمنع تكرار القيم في عمود أو مجموعة أعمدة. ميزة إعطائها اسم: يمكن حذفها لاحقاً وتطبيقها على أكثر من عمود.

🔸 CHECK

-- CHECK على عمود واحد:
CREATE TABLE NamesAges
(
    Name varchar(50),
    Age  int CHECK (Age >= 0)    -- العمر لا يكون سالب
);

-- CHECK على عدة أعمدة:
CREATE TABLE Employee
(
    Name varchar(50),
    a    decimal(12,2),
    b    decimal(12,2),
    CONSTRAINT MyConstraint CHECK (a > b)
);
شرح CHECK يتحقق من شرط معين عند إدخال البيانات. إذا كان الشرط خاطئاً، لن يُدرج السجل.

🔸 PRIMARY KEY

-- مفتاح أساسي من عمود واحد:
CREATE TABLE Books
(
    BookId   int PRIMARY KEY,
    Year     int,
    BookName varchar(50)
);

-- مفتاح أساسي من أكثر من عمود (Composite Key):
CREATE TABLE T1
(
    Pk1  int,
    Pk2  int,
    name varchar(50),
    CONSTRAINT pks PRIMARY KEY (Pk1, Pk2)
);

-- حذف Primary Key في MySQL:
ALTER TABLE MyTable
DROP PRIMARY KEY;
شرح PRIMARY KEY يعني أن هذا العمود (أو مجموعة أعمدة) يحدد كل سجل بشكل فريد. تلقائياً يطبق NOT NULL و UNIQUE.

🔸 FOREIGN KEY

ALTER TABLE Attendance
ADD CONSTRAINT locationid_fk
FOREIGN KEY (LocationId)
REFERENCES Location(LocationId);
شرح السطر 1: نضيف القيد على جدول Attendance.
السطر 2: نسمي القيد locationid_fk.
السطر 3: العمود LocationId في Attendance هو المفتاح الأجنبي.
السطر 4: يشير إلى عمود LocationId في جدول Location (المفتاح الأساسي هناك).

6️⃣ الفهارس (Indexes)

-- إنشاء فهرس:
CREATE INDEX member_name_index
ON MemberDetails (FirstName, LastName);

-- حذف الفهرس في SQL Server:
DROP INDEX MemberDetails.member_name_index;

-- في Oracle / IBM DB2:
DROP INDEX member_name_index;

-- في MySQL:
ALTER TABLE MemberDetails
DROP INDEX member_name_index;

-- في MS Access:
DROP INDEX member_name_index ON MemberDetails;
شرح الـ Index مثل فهرس الكتاب — يجعل البحث أسرع بكثير. يتم إنشاؤه على الأعمدة التي يُبحث فيها كثيراً. كل DBMS له طريقة مختلفة قليلاً لحذفه.
🎨 المحاضرة الثالثة: تصميم قواعد البيانات (Database Design)

🔑 الملخص الشامل

تصميم قاعدة البيانات هو عملية تنظيم البيانات بشكل مناسب. يمر بمراحل: دراسة الجدوى ← تجميع المتطلبات ← التصميم والنمذجة ← التطبيق. أهداف التصميم: الكفاءة، النزاهة، الخصوصية، الأمن، القابلية للتطبيق، المرونة.

1️⃣ مراحل تصميم قاعدة البيانات

1

Feasibility Study — دراسة الجدوى

تحديد الهدف من قاعدة البيانات بوضوح تام قبل البدء في أي شيء.

2

Requirement Collection & Analysis — جمع وتحليل المتطلبات

تحديد البيانات المطلوبة وكيف ستُستخدم. مقابلة المستخدمين بشكل متكرر. مثال: هل للمريض أكثر من طبيب؟

3

Prototyping & Design — النمذجة والتصميم

ثلاث مراحل: التصميم المفاهيمي (Conceptual) ← المنطقي (Logical) ← الفيزيائي (Physical).

4

Implementation — التطبيق

تطوير كود معالجة قاعدة البيانات وتثبيت المحتوى الجديد.

2️⃣ أهداف تصميم قاعدة البيانات

Efficiency — الكفاءة
الأهم. استخدام الحجم الكامل للأجهزة والبرامج المتاحة. لا تأخير للمستخدم عند الوصول.
Integrity — النزاهة
البيانات دقيقة وصحيحة. الحماية من الفساد سواء بالأخطاء البرمجية أو غيرها.
Privacy — الخصوصية
لا وصول غير مصرح به. رصيد عميل لا يُكشف لعميل آخر.
Security — الأمن
الحماية من الفساد الفيزيائي سواء من أعطال الأجهزة أو البرامج أو الوصول غير المصرح.
Implementability — التطبيقية
النموذج المفاهيمي بسيط وفعال. تطبيقات تتفاعل بكفاءة مع قاعدة البيانات.
Flexibility — المرونة
قاعدة البيانات تتكيف مع التغييرات المستقبلية في الأعمال. لا تصميم جامد.

4️⃣ الزيادة والتكرار (Redundancy) وشذوذات البيانات

🇮🇶 العربية

التكرار يعني تخزين نفس المعلومة أكثر من مرة دون فائدة. يؤدي إلى ثلاثة أنواع من الشذوذات:

  • Insertion Anomaly: لا نستطيع إدخال قسم بدون موظف
  • Update Anomaly: لو الموظف له عنوانان مختلفان في السجلات
  • Deletion Anomaly: حذف آخر موظف في قسم يحذف القسم كله
🇬🇧 ENGLISH

Redundancy = storing same data more than once. It causes three types of anomalies:

  • Insertion: Can't add a department without a staff member
  • Update: Employee 519 has different addresses in different records
  • Deletion: Deleting employee 100 removes all sales dept info

5️⃣ Repeating Group — المجموعة المتكررة

تعريف: سمة يمكن أن تحتوي على أكثر من قيمة لنفس المفتاح الأساسي. مثلاً: موظف لديه أكثر من رقم هاتف.
مهم: لا تُسمح Repeating Groups في التصميم العلائقي — كل خلية يجب أن تحتوي قيمة واحدة فقط (Atomic Values).
🔗 المحاضرة الرابعة: الاعتمادية الوظيفية (Functional Dependency)

🔑 الملخص الشامل

الاعتمادية الوظيفية تصف العلاقة بين السمات: A → B تعني "A يُحدد B". أنواعها: كاملة، جزئية، انتقالية. قواعد أرمسترونج (Armstrong's Axioms) تُستخدم للاستنتاج: الانعكاسية، الزيادة، الانتقالية، الاتحاد، التحليل.

1️⃣ مفهوم الاعتمادية الوظيفية

🇮🇶 العربية

A → B تعني: "A يُحدد B". أي إذا عرفنا قيمة A نستطيع معرفة قيمة B بشكل فريد.

  • A يُسمى المُحدِّد (Determinant)
  • B يُسمى المعتمد (Object/Dependent)
  • مثال: StudentID → GPA (رقم الطالب يُحدد معدله)
  • مثال: Child → Mother (الطفل يُحدد أمه)
🇬🇧 ENGLISH

A → B means: "A determines B". Knowing A gives us exactly one value of B.

  • A is the Determinant
  • B is the Dependent
  • Example: StudentID → GPA
  • Trivial FD: A → B is trivial if B ⊆ A

2️⃣ أنواع الاعتمادية

📌 الاعتمادية الكاملة (Full Functional Dependency)

B يعتمد اعتماداً كاملاً على A إذا لم يكن يعتمد على أي جزء من A.
مثال: {Roll No, Subject No} → Grade (الدرجة تعتمد على كليهما معاً، ليس أحدهما فقط)

📌 الاعتمادية الجزئية (Partial Functional Dependency)

B يعتمد على جزء فقط من المفتاح المركب.
مثال: Subject No → Hall Number (رقم القاعة يعتمد على المادة فقط، وليس رقم الطالب أيضاً)

📌 الاعتمادية الانتقالية (Transitive Dependency)

A→B و B→C إذاً A→C (بشكل غير مباشر عبر B).
مثال: Staff Number → Designation → Salary
إذاً: Staff Number → Salary (انتقالية)

📌 المُحدِّد المركب (Compound Determinant)

أكثر من سمة مطلوبة لتحديد سمة أخرى.
مثال: {Internal Mark, External Mark} → Grade

3️⃣ بديهيات أرمسترونج (Armstrong's Axioms)

① الانعكاسية (Reflexivity)
إذا كان Y ⊆ X، إذاً X → Y

أي مجموعة سمات تُحدد أي جزء منها.

مثال: {StudentID, StudentName} → StudentName

② الزيادة (Augmentation)
إذا X → Y، إذاً XZ → YZ

يمكن إضافة سمات إلى طرفي الاعتمادية. لكن لا يجوز إضافتها للطرف الأيمن فقط.

أيضاً: إذا X → Y، إذاً XZ → Y

③ الانتقالية (Transitivity)
إذا X → Y و Y → Z، إذاً X → Z

مثل التوازي: إذا X موازٍ لـ Y وY موازٍ لـ Z، إذاً X موازٍ لـ Z.

④ شبه الانتقالية (Pseudotransitivity)
إذا X → Y و YW → Z، إذاً XW → Z

تعميم للانتقالية (الانتقالية حالة خاصة عندما W فارغة).

⑤ الاتحاد (Union)
إذا X → Y و X → Z، إذاً X → YZ

إذا كان نفس المحدد يُحدد سمتين مختلفتين، يمكن دمجهما.

مثال: PINCODE → City و PINCODE → State، إذاً PINCODE → City, State

⑥ التحليل (Decomposition)
إذا X → YZ، إذاً X → Y و X → Z

عكس الاتحاد. يمكن تفكيك الطرف الأيمن.

مثال: BookID → {Title, Author} إذاً BookID → Title و BookID → Author

⚙️ المحاضرة الخامسة: التطبيع (Normalization)

🔑 الملخص الشامل

التطبيع هو عملية تنظيم الجداول للقضاء على التكرار وشذوذات البيانات. المراحل: UNF → 1NF → 2NF → 3NF → BCNF. في الغالب يكفي الوصول إلى 3NF.

1️⃣ ما هو التطبيع؟

🇮🇶 العربية

التطبيع = إعادة تنظيم الجداول بحيث:

  • كل حقيقة تُخزَّن مرة واحدة فقط (لا تكرار)
  • تجنب شذوذات الإدراج والحذف والتعديل
  • الحفاظ على تكامل البيانات
  • يعتمد على مفهوم الاعتمادية الوظيفية
🇬🇧 ENGLISH

Normalization = organizing tables to:

  • Store each fact only once (no redundancy)
  • Avoid insert/update/delete anomalies
  • Maintain data integrity
  • Based on Functional Dependency concepts

2️⃣ النماذج الطبيعية (Normal Forms)

النموذج الشرط ما يزيله
1NF كل الأعمدة تحتوي قيماً ذرية (Atomic) — لا مجموعات متكررة Repeating Groups
2NF في 1NF + كل سمة غير مفتاحية تعتمد اعتماداً كاملاً على المفتاح الأساسي Partial Dependencies
3NF في 2NF + لا اعتمادية انتقالية Transitive Dependencies
BCNF في 3NF + كل مُحدِّد هو مفتاح مرشح (Candidate Key) حالات استثنائية
5NF اعتماديات معقدة غير شائعة Join Dependencies
ملاحظة: 3NF يُعتبر كافياً في معظم التطبيقات العملية. ما بعده قد ينتج جداول كثيرة جداً وأداء أبطأ.

3️⃣ من UNF إلى 1NF

جدول DEPARTMENT فيه عمود Location يحتوي على أكثر من قيمة (Coimbatore, Chennai). هذا يخرق 1NF.

الحلول الثلاثة:

1

الحل 1: تقسيم العمود (Location1, Location2)

❌ العيب: إذا كان قسم في عشرة أماكن نحتاج 10 أعمدة! كثير من NULL في الأماكن التي فيها قسم واحد.

2

الحل 2: إضافة صفوف جديدة لكل موقع

❌ العيب: تكرار كبير — نفس بيانات القسم تتكرر في كل صف.

3

الحل 3 ✅ (الأفضل): تقسيم الجدول

إنشاء جدولين: DEPARTMENT (DeptID, DeptName) و DEPT_LOCATION (DeptID, Location). هذا هو التطبيع الحقيقي!

4️⃣ من 1NF إلى 2NF

مثال: جدول EMPLOYEE_PROJECT

المشكلة المفتاح المركب: {E_ID, P_ID}
• E_NAME يعتمد على E_ID فقط (اعتمادية جزئية ❌)
• P_NAME يعتمد على P_ID فقط (اعتمادية جزئية ❌)
• TOTAL_TIME يعتمد على {E_ID, P_ID} معاً (اعتمادية كاملة ✅)
الحل نُقسّم إلى 3 جداول:
EMPLOYEE(E_ID, E_NAME) — E_ID → E_NAME
PROJECT(P_ID, P_NAME) — P_ID → P_NAME
HOURS_ASSIGNED(E_ID, P_ID, TOTAL_TIME) — {E_ID,P_ID} → TOTAL_TIME

5️⃣ من 2NF إلى 3NF

مثال: جدول HOSTEL (Roll_No, Building_Name, Fee)

المشكلة Roll_No → Building_Name (مباشر ✅)
Building_Name → Fee (الرسوم تعتمد على المبنى)
Roll_No → Fee (انتقالية ❌ — غير مباشرة عبر Building_Name)
الحل نُقسّم إلى جدولين:
STUDENT_HOSTEL(Roll_No, Building_Name)
BUILDING(Building_Name, Fee)

⚡ المحاضرة السادسة: معالجة المعاملات (Transaction Processing)

🔑 الملخص الشامل

الـ Transaction هو مجموعة عمليات قراءة وكتابة تُنفَّذ كوحدة واحدة. يجب أن تتبع خصائص ACID: Atomicity, Consistency, Isolation, Durability. مشاكل التزامن: WR, RW, WW Conflicts. الحل: Lock-Based Concurrency Control.

1️⃣ المصطلحات الأساسية

Transaction (T)
المعاملة
مجموعة عمليات قراءة وكتابة تُنفذ كوحدة واحدة في قاعدة البيانات.
Read — R(x)
قراءة كائن
قراءة قيمة الكائن x من قاعدة البيانات.
Write — W(x)
كتابة كائن
تعديل وكتابة قيمة الكائن x في قاعدة البيانات.
Commit
إتمام ناجح
إنهاء المعاملة بنجاح وتثبيت كل التغييرات.
Abort
إلغاء
إنهاء المعاملة بفشل والتراجع عن كل التغييرات.
Partial Transaction
معاملة منقوصة
معاملة توقفت في المنتصف — تترك قاعدة البيانات بحالة غير متسقة.

2️⃣ خصائص ACID

A — Atomicity (الذرية)

إما تُنفَّذ المعاملة كاملة أو لا تُنفَّذ إطلاقاً. لا يوجد "نصف معاملة".

أسباب عدم الإتمام: إلغاء ذاتي، انقطاع كهرباء، موقف غير متوقع.

C — Consistency (الاتساق)

المستخدم مسؤول عن ضمان أن المعاملة تترك قاعدة البيانات في حالة متسقة.

مثال: تحويل $100 من حساب A إلى B: يجب خصم نفس المبلغ وإضافته.

I — Isolation (العزل)

المعاملات المتزامنة يجب أن تكون معزولة — تنفيذ أحدها لا يؤثر على الأخرى.

يتحقق بخوارزميات الجدولة مثل Serial Scheduling.

D — Durability (الديمومة)

بعد الـ Commit، التغييرات دائمة ولا يمكن التراجع عنها حتى لو توقف النظام.

يُحقَّق بملفات السجل (Log Files) التي تُسجل كل عمليات الكتابة.

3️⃣ Serial Scheduling والتنفيذ المتداخل

🇮🇶 العربية

Serial Scheduling: تُنفَّذ المعاملات واحدة تلو الأخرى من البداية للنهاية.

Interleaved Execution: تتناوب المعاملات في التنفيذ (تبديل بين T1 وT2 وهكذا). يزيد الكفاءة لكن قد يسبب شذوذات.

🇬🇧 ENGLISH

Serial Scheduling: Transactions execute one after another completely.

Interleaved Execution: Switching execution between transactions. Improves throughput but may cause conflicts.

4️⃣ شذوذات التنفيذ المتداخل (Conflicts)

🔴 WR Conflict — Dirty Read (القراءة القذرة)

التعريف: T2 يقرأ بيانات عدّلها T1 لكن T1 لم يُنهِ (لم يُنفَّذ Commit بعد).
مثال: T1 يخصم $100 من حساب A → T2 يقرأ A ويضيف فائدة → T2 يُنهي → T1 يُكمل ويضيف $100 لـ B. النتيجة غير صحيحة.

🟠 RW Conflict — Unrepeatable Read (القراءة غير المتكررة)

التعريف: T1 يقرأ A، ثم T2 يعدّل A ويُنهي، ثم T1 يقرأ A مجدداً فيجد قيمة مختلفة!
مثال: T1 يتحقق رصيد A ($101) للسحب، T2 يسحب $100 → يبقى $1، T1 يحاول سحب $100 → اضطراب.

🟡 WW Conflict — Lost Update (التحديث الضائع)

التعريف: T1 وT2 يحدثان نفس الكائن.
مثال: T1 يُحدث A=3000، T2 يُحدث A=2000 و B=2000 ويُنهي، T1 يُحدث B=3000. النتيجة: A=2000 و B=3000 (غير متساويين، خرق القيد).

5️⃣ التحكم في التزامن بالأقفال (Lock-Based Concurrency Control)

الـ Lock أنواع:

النوعالوصفالشرط
Shared Lock قفل مشترك — للقراءة. أكثر من معاملة يمكنها الحصول عليه. الكائن غير مقفل حصرياً
Exclusive Lock قفل حصري — للكتابة. معاملة واحدة فقط تملكه. لا أحد يملك قفلاً عليه

Strict Two-Phase Locking (Strict 2PL) — بروتوكول القفل ذو المرحلتين الصارم:

القاعدة 1 للقراءة: طلب Shared Lock. للكتابة: طلب Exclusive Lock.
القاعدة 2 تُحرَّر جميع الأقفال عند انتهاء المعاملة.

6️⃣ الإغلاق المميت (Deadlock)

🇮🇶 العربية

يحدث عندما تنتظر معاملتان أو أكثر بعضها البعض في حلقة مفرغة — لا أحد يُكمل.

طرق التعافي:

  • الكشف من خلال متوسط وقت الانتظار
  • خوارزمية اكتشاف الـ Deadlock
  • إلغاء الكائنات من المعاملة الكبيرة
  • إعادة تشغيل المعاملات المقفولة
🇬🇧 ENGLISH

Occurs when 2+ transactions wait for each other in a circular loop — neither can complete.

Recovery methods:

  • Detection via average waiting time
  • Deadlock detection algorithm
  • Release objects from big transaction
  • Restart locked transactions
